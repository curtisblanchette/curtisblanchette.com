# AGENTS.md — instructions for AI agents implementing cb-site-kit

You are an AI agent helping Curtis Blanchette implement his personal site
using the **cb-site-kit** design system. This file is the binding spec.
If the system says something and you'd rather not — follow the system.

Read this once at the start of any task. Then read `README.md` for the
visual + voice rules.

---

## What the system is

A small CSS-only design system with two stylesheets, a handful of
component HTML snippets, and four page templates.

- **`tokens.css`** — design tokens (CSS custom properties) + global
  defaults + semantic typography utilities. **Drop into the project root
  and link from every page.** Do not edit this file unless the brief
  asks you to retheme.
- **`components.css`** — all component styles, classed `.cb-*`. **Drop
  in and link from every page.** Do not edit this file unless you are
  extending it with a new component (in which case, follow the existing
  naming and add a section comment).
- **`components/*.html`** — one component per file. Each is a complete
  standalone HTML page so it renders by itself. **The component is the
  markup inside `<main class="cb-preview">…</main>` — copy that, not the
  `<head>`.**
- **`templates/*.html`** — full composed pages (home, projects, writing,
  now). **Fork these.** They are the canonical assembly.

## Implementation workflow

When asked to "implement the design" or "build the site":

1. **Identify the target stack.** Look in the user's repo for `package.json`,
   framework manifest files, or ask. Common cases:
   - Static HTML → copy templates verbatim, link the two CSS files.
   - Next.js / Astro / SvelteKit → translate templates into the
     framework's component model, keep classes identical.
   - React component library → port each `cb-*` block into a component
     that renders the same DOM with the same class names. The CSS does
     not need to change.

2. **Copy the kit into the project.** Put `tokens.css`, `components.css`,
   and `fonts/` into the project's public assets directory. Reference
   them from a shared layout / app shell.

3. **Build pages in this order:**
   1. The shared chrome (`cb-nav`, `cb-footer`) as a layout.
   2. `home.html` first — it exercises every component.
   3. `projects.html`, `writing.html`, `now.html`.
   4. A `contact.html` — the kit doesn't ship one (use the home CTA
      block as the body, full-page).

4. **Fill in real content.** The templates ship with placeholder copy
   that follows the voice rules. Replace with Curtis's real content
   but **do not loosen the voice**.

5. **Source real assets.** Replace the inline-SVG project covers with
   real screenshots, sized 4:3.

6. **Test:** every page loads cleanly, every link resolves, dark/light
   alternation is visually correct, the marquee animates, the terminal
   cursor blinks, `prefers-reduced-motion` quiets everything.

## Hard rules

### Do

- **Use the tokens.** Every colour, size, easing, and spacing value in
  new CSS must come from a `var(--*)` defined in `tokens.css`. If a
  token doesn't exist for what you need, add it to `tokens.css` first
  (and name it semantically — `--shadow-card`, not `--shadow-3`).
- **Keep the alternation rule.** Pages alternate light / dark every 2–3
  sections. Apply `class="cb-invert"` to a `<section>` to flip it.
- **Honour sentence case.** Every display heading and page title. Mono
  uppercase is reserved for eyebrows, micros, and pills.
- **Use the existing component classes.** If you find yourself writing
  a new `<div class="my-card">`, stop — there's almost certainly a
  `.cb-*` for it. Search `components.css` first.
- **Preserve the voice.** First-person singular for the engineer.
  Specific over aspirational. No "passionate", no emoji, no "amazing",
  no "rockstar", no "10x", no "We" (he is not a studio).
- **Match Curtis's content shape.** Projects are Helix, Orbit, Ledger
  CLI, Field Notes (templates show four). Essays already have plausible
  titles in the voice — swap to real titles, keep the same length and
  tone. Replace placeholder dates with real ones.

### Do not

- **Do not introduce a new typeface.** Three families, listed in
  README.md. If a heading "wants" a different font, you're using the
  scale wrong.
- **Do not introduce a new colour.** One ink, one paper, one accent.
  Status colours (`--signal-live`) are derived from the accent.
- **Do not invent emoji icons.** The kit has no icons. If a play button
  is genuinely needed, use the native `<video>` chrome.
- **Do not add scroll-driven motion** unless the brief asks. The kit
  is intentionally CSS-only; one terminal blink, one marquee, hover
  states. That's it.
- **Do not add a "philosophy" or "about" section** to subpages. If
  the user asks, push back: every page should earn its place. The home
  page's intro paragraph is the only place a manifesto is allowed.
- **Do not fall back to Inter / system-ui** for display. The brand
  fonts must be self-hosted (they are in `fonts/`).
- **Do not pad with filler content.** No "Featured by", no
  "Recent companies I've worked with" logos unless Curtis explicitly
  asks for them. One thousand no's for every yes.

### Edits to the kit itself

If a brief requires a new component:

1. **Add the CSS** to `components.css`, at the bottom, with a header
   comment that matches the existing pattern.
2. **Add an HTML snippet** in `components/<your-component>.html` using
   the same preview shell as the others.
3. **Document it** in `README.md`'s "Components cheat sheet" table.
4. **Update this AGENTS.md** if there's a usage rule that a future
   agent needs.

If a brief requires retoning the accent permanently, **add a new theme
class** in `tokens.css` (next to `cb-theme-sodium`, `cb-theme-amber`,
etc.) rather than changing the `--accent` default.

## Translating to frameworks

The component HTML is intentionally framework-agnostic. To port to React,
the mapping is:

```jsx
// HTML
<a href="#" class="cb-project-row">
  <span class="cb-project-row__num">01</span>
  <h3 class="cb-project-row__title">Helix — <em>a platform</em>…</h3>
  ...
</a>

// React
function ProjectRow({ num, title, italic, meta, href }) {
  return (
    <a href={href} className="cb-project-row">
      <span className="cb-project-row__num">{num}</span>
      <h3 className="cb-project-row__title">{title} <em>{italic}</em></h3>
      <span className="cb-project-row__meta">{meta}</span>
      <span className="cb-project-row__arrow">↗</span>
    </a>
  );
}
```

Note: `class` → `className`. Inner `<em>` stays an `<em>` — it gets
the PP Eiko italic flourish via the existing CSS rule. **Do not move
the italic styling into a React prop or a Tailwind class.**

## What "done" looks like

- Every page renders cleanly with no console errors and no broken
  assets.
- Reduced motion mode (`prefers-reduced-motion: reduce`) silences all
  animation (verify: the marquee freezes, the cursor stops blinking).
- The home page passes the **glance test**: at 2× zoom-out, you should
  see (a) Curtis's name big, (b) a clear list of projects, (c) one
  pull quote, (d) the Now block, (e) the CTA. No ambiguity about what
  this site is.
- The accent retones correctly: add `class="cb-theme-amber"` to
  `<html>` and the green disappears from every component.
- Lighthouse: a11y ≥ 95, no missing alt text on real screenshots, focus
  rings on every interactive element.
